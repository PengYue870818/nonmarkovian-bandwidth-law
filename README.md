# nonmarkovian-bandwidth-law
Two Python programs for non‑Markovian QBL. QBL-CPV performs cross platform single-window validation of lever span identity across qubit platforms under short and long memory. QBL-DPBV implements geometry memory separation where the instrument locked spatial slope is invariant and memory compresses the log bandwidth. 

This archive contains two Python programs:

## 1. QBL-DPBV.py
The program "QBL-DPBV" implements a complete single‑window workflow to validate geometry–memory separation in open quantum dynamics under a practical Quantum Bandwidth Law. It constructs a common frequency ruler from a rectangular time window, selects an operational band by a coverage rule, computes a logarithmic span and an in‑band gain on that same ruler, and checks that the leading response collapses onto the span (or the gain times the span). The instrument geometry is encoded once in a geometry‑locked factor that does not change with memory.

"QBL-DPBV" runs a complete single‑window analysis to validate geometry–memory separation under a practical Quantum Bandwidth Law. It builds a common frequency ruler from a rectangular window, selects an operational band by a coverage rule, computes the logarithmic span and (optionally) an in‑band gain on that same ruler, and produces both figures and CSV tables that document edge selection, collapse checks, geometry invariance, remainder diagnostics, master curves, time‑domain versus frequency‑domain consistency, a step‑PSD counterexample, convergence checks, and an automated validation report.

The minimum system requirements are modest. You can run on Linux, macOS, or Windows (64‑bit) with Python 3.8 or newer. A modern multi‑core CPU and 4 GB of RAM are sufficient for the default run, though 8 GB is recommended for extended scans or very fine spectral grids. Ensure several hundred megabytes of free disk space for the figures and CSV outputs.

Only three dependencies are required: NumPy, SciPy, and Matplotlib. After creating or activating your Python environment, install them with a single command such as "pip install numpy scipy matplotlib". The script forces a non‑interactive plotting backend so it runs cleanly on headless servers.

Running the program is straightforward. Place the script in a working folder and launch it with "python your_script.py". It automatically creates an output directory named "figs_nm_mmsis" and writes all figures (PNG) and tables (CSV) there. No command‑line arguments are required. The default configuration carries out short‑ and long‑memory validations, generates the counterexample that highlights why edge‑based selection matters, and produces an integrated pass/fail report.

Results are saved in the "figs_nm_mmsis" directory. A central summary table records physical scales, the geometry factor, band limits and caps, flatness diagnostics, and runtime information. Collapse figures compare the normalized response to the predicted linear behavior, while the geometry‑invariance figure shows the geometry factor versus memory time. Edge‑selection panels visualize the operational band, and time‑domain panels compare the triangle‑weighted time integral to the gain‑times‑span product. Every figure has a matching CSV with exactly the numbers plotted, so you can re‑plot without rerunning the heavy computations. An automated validation report aggregates pass and fail flags for inequalities, short‑ and long‑memory edge laws, geometry locking, remainders, time‑versus‑frequency agreement, and convergence.

Basic customization is done by editing the configuration object near the top of the script before running. You can change the output directory, the window length that sets the frequency ruler, the coverage fraction used for edge selection, and the number of logarithmic frequency samples that controls spectral resolution. You can also enable or disable susceptibility in the gain path, choose a flat, Gaussian, or Lorentzian model, and provide your own memory‑time lists for short and long runs. After editing, simply rerun the script. For a more exhaustive test suite that includes extended failure‑phase diagrams and multi‑window or self‑energy options, call the extended runner inside the main block and expect longer runtime and larger outputs.

If runtime or memory use becomes an issue, reduce the number of log‑frequency samples or shorten the memory‑time lists; if you need tighter accuracy, increase the spectral grid or enable the built‑in convergence checks to confirm stability. If the operational band appears too narrow, consider lengthening the window within the strictly monotone main lobe or relaxing instrument caps; if in‑band dispersion is strong due to susceptibility tilt, enable the gain path and optionally contract the band until the band‑averaged gain is near unity.

## 2. QBL-CPV.py
The program "QBL-CPV" performs a cross‑platform, single‑window analysis of spectra to test geometry–memory separation and bandwidth laws on a common frequency ruler. It reads power spectral density data from CSV files when available, synthesizes a reference spectrum when data are missing, automatically selects an operational band per platform, and generates collapse plots, operational range summaries, pass/fail reports, and robustness diagnostics. All figures and tables are written to a single output directory so you can inspect results without additional post‑processing.

You can run the program on Linux, macOS, or Windows (64‑bit) with Python 3.8 or newer. A modern multi‑core CPU and 4 GB of RAM are sufficient for the default run; for large grids or extended robustness analyses, 8 GB is recommended. Disk usage is modest; a few hundred megabytes of free space will comfortably hold the figures and CSV outputs.

The only required Python packages are NumPy, SciPy, and Matplotlib. After creating or activating your Python environment, install them with a single command such as: pip install numpy scipy matplotlib. The script forces a non‑interactive plotting backend so it runs cleanly on headless servers.

To use the program, place the script in a working folder. If you have platform PSD files, put them in the same folder with the expected names and column headers: FluxQubit_Bylander2011_PSD.csv with columns f_Hz and S_phi_per_Hz, NV_Diamond_Romach2015_PSD.csv with columns f_Hz and S_freqPSD_per_Hz, and SiGe_QD_Connors2020_Sepsilon_vs_f.csv with columns f_Hz and S_epsilon_eV2_per_Hz. If any of these files are missing, the script will still run using a synthetic reference spectrum for the remaining platform. Launch the program with: python your_script.py. No command‑line arguments are needed; defaults are chosen to produce a complete output set.

Results are written to the directory cross_platform_out, which is created automatically. You will find collapse figures and tables for short and long memory, operational “radiation range” plots that show each platform’s band and active edge selector, and a concise pass/fail report that aggregates inequality gaps, collapse slopes and goodness‑of‑fit, and per‑platform neutrality checks. The folder also includes validation and robustness outputs such as coverage controls at alternate coverage fractions, stability summaries under small window and spectrum perturbations, and jitter analyses with log‑normal noise injected into the spectra. Every figure has a matching CSV file containing the exact numbers plotted, allowing you to re‑plot results or integrate them into other workflows without rerunning heavy computations.

Basic customization is done by editing a few variables near the main section before running. You can change the output directory, initial window lengths per platform, the coverage fraction for edge selection, the instrument cap, whether to use purely theoretical shapes, and the file names or column headers for your PSD inputs. The default settings are chosen to reproduce a full cross‑platform comparison as long as the PSD files are present; with no files, the run still completes using the synthetic spectrum so the workflow remains demonstrable end‑to‑end.
